# 中国行政区划知识库

这是一个面向中国行政区划的中文知识型开源项目。它不再以代码工具为中心，而是以文稿、政策、来源、历史沿革和核心数据为中心，目标是把“行政区划代码”这件事整理成一套可持续维护、可追溯、可被大模型直接消费的知识底座。

本仓库当前保留的核心数据文件是 `data/distict.json`，它来源于文档中提到的官方地址下载结果，是项目最重要的数据资产。

## 项目愿景

中国行政区划不是一张静态表，而是一套持续演进的公共知识体系。一个地区的名称可能调整，隶属关系可能变化，代码可能沿用，也可能重编，政策口径也会随着主管部门发布机制而更新。

我希望把这个项目做成三件事：

1. 让行政区划代码的来源更清楚。
2. 让区划变化的历史更完整。
3. 让大模型能够基于可信知识，稳定生成 JS、SQL、CSV、JSON、文档说明等多种结果。

这不是一个单纯的数据仓库，而是一个面向中文区划知识生产的公共基础设施。

## 政策变化

行政区划代码的发布和查询口径，近年已经出现了明显变化。

### 1. 民政部口径正在成为主线

按照《行政区划代码管理办法》的公开精神，行政区划代码由民政部门确定、发布。过去常见的年度页面和旧栏目，正在逐步让位于国家地名信息库与行政区划查询入口。

这意味着：

- 最新的县级以上行政区划代码，应以民政部相关公开入口为准。
- 乡级及以下信息，不能简单依赖单一旧页面，应结合省级民政部门公告与其他官方来源。
- 任何“最新”数据，都必须明确数据时点和来源页。

### 2. 国家统计局口径更偏向统计用途

统计局公开的“统计用区划和城乡划分代码”，长期服务于统计业务，而不等同于行政区划建制本身。

这类数据适合用来理解统计口径、城乡属性和编码规则，但不应被当作行政区划的唯一权威来源。尤其在最新具体代码的公开层面，应注意它和民政部口径的边界已经更清晰。

### 3. 地图厂商数据只能做兼容参考

百度、高德、腾讯等地图平台提供的行政区划码，适合做产品兼容和地址映射参考，但不应作为权威历史依据。

本项目的原则很简单：

- 民政部口径优先。
- 统计口径作为补充。
- 地图厂商口径只用于兼容。

## 这个项目最终要解决什么

我希望它最终能回答这些问题：

- 某个地区的行政区划代码是什么，当前有效吗。
- 这个地区的代码历史怎么变的，为什么变。
- 某一年底的官方快照是什么。
- 某个县改名、撤县设区、街道调整后，代码如何承接。
- 如何基于同一套知识源，自动生成前端联动数据、数据库初始化脚本、接口文档和说明文稿。

## 对 AI 的要求

这个项目不是“把数据给模型看一眼”那么简单，而是要让 AI 在明确来源、明确时点、明确口径的前提下工作。

所以我希望未来的知识库具备这些能力：

- 让模型知道哪些是行政区划，哪些是统计区划。
- 让模型知道每条结论来自哪里。
- 让模型知道哪个版本、哪个时点有效。
- 让模型能直接产出可用代码，而不是只会说概念。

换句话说，这个项目的目标不是追求热闹，而是把中国行政区划这套公共知识，整理成真正能长期使用的基础设施。

## 说明

本仓库已去除原有爬虫与工具代码，保留文稿与核心数据。后续内容将围绕知识整理、政策变化、历史沿革、来源归档和大模型可用性展开。
